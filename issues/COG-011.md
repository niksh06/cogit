# COG-011: "Why not plain Git?" positioning ADR

Priority: P0 (product risk, blocks external messaging)

## Problem

The single most likely objection to Cogit is: "store JSON facts in a
directory and use Git itself — immutable history, branches, reflog, and
blame for free." No current document answers it, and the answer decides
whether Cogit is a product or a convention on top of Git.

## Task

Write ADR-0011 that either:

1. justifies the standalone implementation, or
2. redefines Cogit as a porcelain over Git plumbing.

The honest argument inventory for standalone (to be tested, not assumed):

- typed schema validation at write time (Git stores any bytes);
- set-based, claim-aware merge (Git merges text lines and would produce
  meaningless conflicts on JSON sets);
- first-introducer blame over fact identity (Git blame is line-oriented,
  last-modifier);
- agent guardrails in the storage layer: dirty-index checkout block,
  secrets rejection, no-prune default;
- one `.cogit/` with no packfiles/gc surprises — inspectability contract.

Counter-arguments to address: Git's maturity, ecosystem, hosting; custom
merge drivers; `git notes`; trailer conventions.

## Acceptance criteria

- ADR-0011 exists with an explicit decision and trade-off table.
- README references the ADR in one sentence.
- If decision is "porcelain over Git", COG-013 (Rust port) is re-scoped.
