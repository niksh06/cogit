# COG-020: Conflict resolution memory (rerere)

Priority: P1 (US-019, roadmap Phase 4; closes OQ-008)

## Problem

Long-lived hypothesis branches produce the same claim-family conflict on
every merge. Asking for the same arbitration repeatedly wastes agent turns
and invites inconsistent answers.

## Decisions (conflict fingerprint, closes OQ-008)

- Fingerprint = SHA-256 of the canonical JSON of the normalized conflict
  shape:

  ```json
  {
    "claim": "<negation-group root claim id>",
    "sides": [["<assertion ids>"], ["<assertion ids>"]],
    "base": ["<assertion ids>"]
  }
  ```

  Both sides are sorted internally and the two side-lists are sorted
  against each other, so merge ORIENTATION does not matter: A-into-B and
  B-into-A of the same rivalry share one fingerprint. `base` is included:
  the same rival pair over a different base is a different conflict.
- Store: `.cogit/rerere.json`, a mutable LOCAL preference file (analogous
  to git's `rr-cache`) — not immutable history, never synced, written via
  tmp+rename.
- Recording: every successful `cogit resolve` (manual or suggested)
  records `{fingerprint -> keep-or-drop}` automatically.
- Suggesting: `merge` attaches `"suggestion"` and `"fingerprint"` to
  matching conflict entries in the index and prints them. Suggestions are
  NEVER auto-applied — the agent must run `cogit resolve <claim>
  --suggested`, keeping the resolution visible before commit (US-019,
  ADR-0009 conservatism).
- Forgetting: `cogit rerere --forget <claim-or-fingerprint>`; `cogit
  rerere` lists stored resolutions.

## Acceptance criteria

- Same conflict shape yields the same fingerprint regardless of merge
  direction; different base yields a different fingerprint.
- A repeated merge surfaces the stored suggestion; `resolve --suggested`
  applies it; commit succeeds with the expected mindset.
- Forgotten resolutions stop being suggested.
- Nothing is ever resolved without an explicit `resolve` invocation.
