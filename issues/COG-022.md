# COG-022: `count-objects` and maintenance thresholds

Priority: P1 (US-021, roadmap Phase 5, ADR-0006)

## Problem

ADR-0006 defers every performance layer until measured pressure — but
nothing measures pressure. Optimization decisions need data from real
repositories.

## Decisions

- `cogit count-objects [--json]` reports: loose objects by type, total disk
  size, corrupt-object count, branch and anchor counts, reflog entries and
  bytes, stale tmp files.
- Type counting reads object headers only (decompress + header parse), not
  full schema validation — this is a metrics scan, not `verify`.
- Warnings when thresholds are exceeded; defaults follow ADR-0006
  (5000 loose objects, 200 refs, 10000 reflog entries), overridable via a
  `[maintenance]` section in `.cogit/config`.
- The command never mutates the repository.
- `gc --auto` from US-022 is DEFERRED until a compaction mechanism exists;
  a gc command that can only no-op would be noise.

## Acceptance criteria

- Counts match a freshly built fixture repository.
- Threshold override in config triggers warnings.
- CLI contract documents the command; US-022 rows stay open in the backlog.
