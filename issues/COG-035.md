# COG-035: Atomic micro-commit and index locking

Priority: P0 (shared-journal experiment; supersedes and closes COG-033)

## Problem (COG-034 finding, 2026-07-03)

The owner's shared-journal experiment (one `~/.cogit-journal` for all
sessions and subagents) had to encode a WORKAROUND into the agent
constitution: parallel agents may only `add_fact(commit=true)`, never
branch/checkout — because HEAD and `index.json` are shared. Worse, even
that workaround is unsound: `index.json` is read-modify-write without a
lock, so two concurrent micro-commits can interleave (agent B's staged
fact swallowed into agent A's thought; B's commit then fails).

## Decisions

1. **Index lockfile** (COG-033 option 1): every index read-modify-write
   acquires `.cogit/index.json.lock` (O_EXCL create, ~2 s retry window,
   then the concurrency exit code with a stale-lock hint). Lock ordering:
   the index lock may be held while taking ref locks, never the reverse —
   ref locks are always released before index writes.
2. **Atomic micro-commit**: `add_fact --commit` / MCP `commit=true` stops
   going through the shared index entirely. The new mindset is composed
   directly from the parent thought's mindset plus the new assertion, and
   published with the ref old-target check in a retry loop (on concurrent
   movement, recompute from the new HEAD, up to 5 attempts). Parallel
   micro-commits are safe BY CONSTRUCTION; the constitution workaround can
   shrink to "parallel agents: use micro-commits".
3. Micro-commit still refuses a dirty index (it must not invalidate an
   in-progress staging session's base) and refuses contradictions
   (invariant 25) — refutation requires the staged remove+commit flow.
4. A fact already active at HEAD micro-commits to a no-op:
   `already_active: true`, no empty thought is created.

## Acceptance criteria

- Thread-parallel test: 2 writers x N micro-commits on one repository all
  succeed; the final mindset holds all facts; `verify` is healthy.
- Lock contention surfaces as the concurrency exit code with a
  remediation hint; recovery playbook documents stale locks.
- Both runtimes; interop unchanged (lock protocol is shared).
