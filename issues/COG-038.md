# COG-038: Read-only web viewer for a journal

Priority: P1 (owner request, COG-034 usage phase)

## Problem

The only way to inspect a live shared journal is CLI porcelain or MCP
calls — fine for agents, poor for the human auditor. Watching the
thought DAG, active beliefs, blame chains, and negation families needs a
visual surface. The concern "isn't a UI spying on the model?" is
answered by the product's own boundary: Cogit stores committed beliefs,
never raw reasoning (non-goals, cookbook Rule 7). A viewer is a lens
over data the CLI already exposes — it creates no new access.

## Decisions

- **Read-only by construction** (ADR-0009 analogy with the MCP surface):
  the viewer performs no mutating repository calls at all. Write
  operations from a browser would also corrupt provenance (no honest
  `actor`), so they are out of scope, not merely deferred.
- Integration, not CLI: `prototype/integrations/web_viewer.py`, zero
  dependencies (stdlib `http.server`), same placement rule as the MCP
  server. No Rust parity required for integrations.
- One data endpoint: `GET /api/state` returns the full decoded snapshot
  (graph, beliefs, anchors, branches, recap, per-thought fact deltas,
  first-introducer per assertion, annotations). The page fetches once
  and renders; live mode polls. This keeps the server surface minimal
  and makes the snapshot export byte-identical in behavior.
- Snapshot mode: `--snapshot out.html` embeds the same state JSON into
  the page (`window.COGIT_STATE`) producing one self-contained file —
  shareable, artifact-friendly, no server.
- Binds `127.0.0.1` by default; no auth in MVP, so non-local binds are
  an explicit `--host` decision. Threat note: the journal may contain
  project-sensitive beliefs; localhost default keeps exposure at the
  level of the CLI itself.
- Introducer map is derived in the viewer from public porcelain only
  (per-thought `facts()` + parents), one oldest-first pass — no private
  `Repository` internals, no new core API.

## Acceptance criteria

- `python3 web_viewer.py --repo <path>` serves `/` (HTML) and
  `/api/state` (JSON) with: thought DAG (all branch tips + HEAD, topo
  order), branch/anchor badges, active facts of HEAD with
  subject/predicate/project filtering client-side, per-thought
  added/removed fact rows, per-fact introducer (blame), annotations,
  no-arg recap block.
- `--snapshot out.html` writes a self-contained page with identical
  content; `</script>` injection via fact text is impossible (JSON is
  `<`-escaped).
- Unknown paths return JSON 404; repository errors surface as JSON 4xx,
  never tracebacks; server never writes to the repository.
- Tests cover: state shape on a forked history, introducer correctness,
  HTML serving, snapshot embedding, 404.
