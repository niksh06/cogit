# COG-040: Render negation explicitly in agent-facing output

Priority: P1 (COG-039 finding F1 — measured comprehension failure)

## Problem

A negating claim carries the SAME subject/predicate/object as the claim
it negates, plus a `negates` pointer. In `facts` output (CLI text, JSON
rows, MCP) such a row is visually indistinguishable from an active
belief in that value. In the COG-039 benchmark 3 of 4 context-free
journal readers misread refuted beliefs as still-active ("50 remains the
current active belief", "negation implies True") — while the markdown
line "refuted ... no longer believed" confused nobody. The best-audited
medium lost to prose on exactly the semantics it exists to guarantee.

## Decisions

- `facts`/`show` text output prefixes negating rows with `NOT ` on the
  object column and appends `(negates <short-id>)`.
- JSON rows keep `negates` and gain `"negation": true` (derived, no
  format change to stored objects — porcelain-level only).
- Web viewer belief table renders the ⊘ chip on the negating row itself
  (today the chip logic exists but the row still reads `s p = o`), and
  the fact detail panel words it as "asserts NOT (s p = o)".
- MCP `facts` tool description explains the semantics in one line so a
  reader model does not need to guess.
- Cookbook Rule 5 gets a sentence: a negation asserts the proposition is
  FALSE; it does not assert any replacement value.
- Both runtimes (Python + Rust CLI parity); interop step extended with a
  negation-rendering check.

## Acceptance criteria

- A refuted-belief probe answered from `facts` output alone cannot be
  parsed as an active belief in the negated value (text says NOT; JSON
  says negation: true).
- Tests in both runtimes cover the rendering; interop step green.
