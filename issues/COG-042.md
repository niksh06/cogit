# COG-042: `cogit dump` — one-call reader surface for agents

Priority: P1 (COG-039 F2, reinforced by COG-041: journal readers burn
10-18 porcelain calls to assemble one picture)

## Problem

Porcelain answers one question per call (`facts`, then `blame-fact` per
row, then `recap`, then `log`...). Both benchmark runs show a
context-free reader needs 8-18 tool calls to reconstruct the belief
state the web viewer already assembles in ONE `/api/state` document
(COG-038). Reading cost, not accuracy, is where the journal loses to a
single markdown Read.

## Decisions (proposed)

- `cogit dump [--ref R] [--project P] [--since <anchor>] --json`: one
  document with active fact rows (decoded, negation-explicit), introducer
  per assertion, anchors, branches, recent thought log, and the recap
  block against the newest (or given) anchor — the `/api/state` shape,
  reusing the same public porcelain internally.
- MCP tool `dump` mirroring it, described as THE way for an agent to
  re-anchor in one call (recap stays for the delta-only case).
- Both runtimes + interop step; size guard: `--limit-log N` (default 50)
  so the document stays bounded on long histories.

## Acceptance criteria

- A context-free reader can answer all COG-039 probe classes from ONE
  `dump` call (add a benchmark mode exercising this).
- Python + Rust parity, interop step, tests both runtimes.
