# COG-047: `cogit lint` — mechanical cookbook checks

Priority: P1 (first real-world case: COG-045's live run found a
neighbour project with every claim family stuck at one revision)

## Problem

The object format cannot reject prose objects, date-stamped subjects or
blob qualifiers — and those silently disable the features the journal
exists for: claim identity, supersede chains, volatility, calibration,
negation comparability. Guidance lives in the cookbook; nothing checks
it.

## Decisions

- Lint the ACTIVE beliefs of a ref (history lint later if needed):
  R1 multi-proposition objects (`;`/` and `), R2 prose objects (>12
  words or >100 chars), R3 unstable ids (dates/whitespace in
  subject/predicate, overlong subjects), R4 band mismatches (tool <9000,
  agent >=9800), R6 blob qualifiers (>80 chars), R8 missing project
  qualifier in a shared journal. Severities warn/info; findings carry
  rule ids and per-rule summary.
- Placement: integrations (`lint.py` CLI + MCP tool `lint`), read-only,
  same staging as analytics — core-CLI promotion after the rule set
  stabilizes. `--strict` exits 1 on warnings (CI-friendly).
- Heuristics are intentionally simple and named after cookbook rules so
  a finding is a link into the doc, not a mystery.

## Acceptance criteria

- Fixture repo with one violation per rule yields exactly those rule
  ids; clean fact yields none; project filter scopes the run; empty
  repo is clean. MCP tool listed and callable.
- Live-journal run documented (finding counts by rule).
