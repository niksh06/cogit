# COG-048: Secret heuristic false-positive on filesystem paths

Priority: P1 (blocks legitimate use: a report path in an object was
rejected as a "high-entropy token" during live journaling 2026-07-05)

## Problem

`reject_suspected_secrets` (COG-023 entropy heuristic) flagged
`~/Reports/projects/<proj>/reference/<slug>-2026-07-05.md` (70 chars) as
a secret. Filesystem paths are common legitimate belief values (memo
pointers, artifact locations) and are structurally distinguishable from
tokens: multiple `/` separators, dictionary-word segments, known
extensions.

## Decisions (proposed)

- Add a path guard to the entropy check in BOTH runtimes: a candidate
  token containing >= 2 `/` separators whose segments are short
  word-like chunks (or that starts with `/`, `./`, `~/`) is not a
  secret candidate. Credentials-in-URL detection must stay intact
  (`scheme://user:pass@host` keeps rejecting).
- Regression tests both runtimes: report-hub style paths accepted,
  AWS/URL-cred/ghp shapes still rejected; interop step untouched
  (vectors unaffected — heuristic is write-side only).

## Acceptance criteria

- The exact live-failure string is accepted; existing `test_secrets`
  matrix stays green in Python and Rust.
