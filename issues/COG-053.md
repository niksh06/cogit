# COG-053: recap --project — scoped resume on a shared journal

Priority: P1 (external review from a sibling session: no-arg recap on the
recommended resume path returned ~98 KB of cross-project delta and blew
the reader's token budget; reproduced at 110 KB)

## Review assessment (what we verified before fixing)

- CONFIRMED: `repo.recap` had no project parameter; MCP recap schema was
  from/to only; asymmetry with `facts`/`dump` real; COG-037 scoped the
  qualifier filter to `facts` only. Noise reproduced: no-arg
  `recap --json` on the live shared journal = 110,497 chars.
- REFUTED: "orphan `--project` flag declared in recap's CLI" — the recap
  parser declared no such flag at all; nothing was silently dead. The
  reviewer most likely read the adjacent `dump` parser.
- GAP in the proposed fix: filtering only added/removed rows would leave
  the cross-project `thoughts` list (~65 alien messages) — the fix must
  scope thoughts too.

## Decisions

- `recap(source, target, project=None)` in both runtimes: added/removed
  rows filtered by the project qualifier AND the thought list keeps only
  thoughts that CHANGED that project's beliefs (per-thought mindset delta
  vs parents, computed over the bounded from..to window with caches).
- CLI `--project` on recap (now actually declared and wired); MCP recap
  schema gains `project`; `dump` forwards its own project into the recap
  block (the session-start digest header becomes project-scoped as a
  side effect).
- Result carries `"project"` so a reader knows the scope it received.

## Acceptance criteria

- On a two-project fixture: scoped recap returns only the project's rows
  and only the thoughts that touched them; unknown project yields empty
  lists, not errors. Tests in both runtimes; interop step compares
  scoped shapes across runtimes.
