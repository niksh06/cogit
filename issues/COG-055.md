# COG-055: Transactional MCP `record` + protocol hardening

Priority: P0 (verified live-use correctness defect: a failed batch can leave
partial staged state, and malformed tool arguments can escape the MCP error
boundary)

## Problem

`tool_record` mutates the shared index one removal/fact at a time and commits
only after the loop. If a later item is invalid, earlier items remain staged.
The caller receives an exception even though repository state changed.

Verified on 2026-07-10 with a temporary repository: a two-fact `record` whose
second fact omitted `kind` raised `KeyError('kind')` and left the first
assertion in `status.staged`. `handle()` catches only `CogitError`, so this
`KeyError` also escapes the normal `isError` response and can terminate the
stdio server loop.

This violates the advertised contract: “N facts + optional removals -> one
thought”. It is also the unsafe foundation for any lifecycle convenience
operation.

## Decisions (proposed)

- Validate the complete MCP payload and prepare every removal/fact before the
  first mutable repository operation. Missing/unknown fields and invalid types
  are JSON-RPC invalid-params errors, not Python exceptions.
- Apply a successful batch under one repository transaction boundary: exactly
  one thought becomes reachable, or HEAD and the index remain unchanged.
- Refuse `record` when the shared index is already dirty. A batch must not
  absorb or invalidate another staging session.
- Convert expected validation/user failures into MCP `isError` responses.
  Unexpected `Exception` values become a sanitized internal-error response and
  are logged to stderr; they do not kill the next request. Never catch
  `BaseException` and never echo secrets or the raw payload.
- Immutable objects that are not reachable after a failed operation must not
  be treated as a successful partial record. Prefer validate-before-write; if
  unavoidable, document and count unreachable objects as maintenance debris.

## Non-goals

- Inferring which active belief a new fact should supersede.
- Automatically repairing a dirty index.
- Changing the object-format identity rules.

## Resolution (2026-07-10)

Both defects reproduced exactly as filed (two-fact batch, second missing
`kind`: first assertion left staged; `KeyError` escaped `handle()` — in
`main()` that kills the process, not just the request). Fix went further
than the proposed index transaction: `record` no longer touches the
shared index at all. New `Repository.micro_commit_batch(docs, removals,
message)` extends the COG-035 CAS/retry model to N facts + M removals —
validate + write immutable objects first, then publish
`mindset = parent - removed + added` as ONE thought under a ref
old-target check. All-or-nothing by construction, invariant-25 checks
preserved (synthetic batch index feeds `_check_negation_consistency`),
removals must target active assertions (stale target -> clean error),
dirty index refused BEFORE any mutation so a concurrent staging session
survives untouched. MCP side: full payload shape validation up front,
and `handle()` gained a sanitized `except Exception` boundary (type name
only, traceback to stderr, no payload echo). This primitive is the
foundation COG-056 asked for.

## Acceptance criteria

- A batch with a valid first fact and invalid second fact returns `isError`;
  HEAD and index bytes match the pre-call snapshot, and a subsequent `status`
  call through the same MCP process succeeds.
- The same all-or-nothing guarantee holds when a later removal is invalid or a
  fact fails secret/premise/schema validation.
- Unknown fields, missing required fields, wrong JSON types and malformed
  removal entries never escape `handle()`.
- A dirty index is rejected before mutation with an actionable error.
- A valid batch with N facts and M removals produces exactly one thought and
  clears the index.
- Tests cover direct handler calls and a real stdio subprocess that continues
  serving after malformed input. Existing MCP workflow tests stay green.
