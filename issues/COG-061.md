# COG-061: build_state complexity — quadratic object reads at scale

Priority: P2 (entry criterion MET by measurement, urgency low: ETag/304
from COG-060 already removed the unchanged-poll waste; this bites on the
FIRST load and on every real change of a large journal)

## Measured problem (2026-07-10, COG-060 benchmark)

`web_viewer.build_state` decodes the fact rows of EVERY thought's full
mindset (`repo.facts(oid)` per thought) to compute per-thought deltas and
first-introducers: O(N thoughts × M active assertions) object reads with
zlib decompression, no cache.

| Scale | Thoughts | Rebuild |
|---|---|---|
| 1x (live) | 185 | 1.38 s |
| 10x | 1850 | 145.7 s (quadratic, not the ~14 s linear growth) |

Artifact: `~/Reports/projects/cogit/research/2026-07-10-cog-060-viewer-poll-benchmark.md`.

## Decisions (proposed)

- Replace per-thought `facts(oid)` with mindset assertion-SET diffs
  (ids only) and decode each assertion row ONCE across the whole build —
  a memoized row reader turns the pass into O(objects), matching what
  `dump` and `analytics` already pay.
- Keep the viewer on public porcelain: if set-level access needs a public
  surface, add it explicitly (e.g. `diff` already returns id sets) rather
  than reaching into Repository internals.
- Re-run the COG-060 benchmark after the fix; acceptance is linear-ish
  scaling (10x journal within ~10-15x of 1x rebuild, not 100x).

## Non-goals

- Daemons, background indexes, or any cache whose correctness depends on
  invalidation. (The resolution DOES memoize immutable content-addressed
  objects in-process — that is not the thing this non-goal forbids: an
  oid's decoded value can never change, so there is nothing to
  invalidate. Cross-process caches remain out.)
- Changing the /api/state document shape.

## Resolution (2026-07-11)

Two changes in `build_state`:

1. Per-thought `repo.facts(oid)` (decode every active row for every
   thought) replaced with mindset ID-SETS + one row decode per unique
   assertion (`fact_row`/`mindset_assertions` made public porcelain).
2. Immutable-object caches keyed by repo path: content addressing makes
   them trivially correct; incremental rebuilds pay only for new objects.

Re-measured (same 10x synthetic, 2510 thoughts):

| | before | after (cold) | after (incremental) |
|---|---|---|---|
| 1x (185-251 thoughts) | 1383 ms | 241 ms | ~tens of ms |
| 10x | 145 703 ms | 7 100-13 500 ms | ~3 000 ms |

Cold ratio vs 1x is ~30x, not the hoped 10-15x: the residual term is
inherent — every thought's mindset OBJECT lists all M active ids, so
reading N mindsets is O(N x M) bytes; the incremental path removes the
I/O but Python set-diffs across all thoughts remain O(N x M) element
ops. Good enough: the unchanged-poll path is 304 (COG-060), post-change
container rebuilds ride the cache, and a 2500-thought single-page DAG
needs UI pagination long before this needs more speed. Revisit only
with a measured complaint.
