# COG-062: hook off the shared index — parallel sessions stop colliding

Priority: P1 (field report from the aleph marathon: two parallel
sessions with the hook enabled trip dirty-index refusals mid-turn and
absorb each other's staged captures under the wrong author)

## Root cause

The hook was the last writer using the shared index: PostToolUse staged
each event (`add_fact` + supersede `remove_fact`), Stop committed. While
any turn had captured events, the index was dirty — so every
`add_fact(commit=true)` / `record` / lifecycle call from a PARALLEL
session hit the dirty-index refusal, and whichever session's Stop ran
first committed BOTH sessions' staged events under its own author
(breaking Rule 10 attribution).

## Resolution (shipped with this ticket)

- PostToolUse appends events to `.cogit/hook-buffers/<session>.jsonl` —
  a per-session buffer; the shared index is never touched.
- Stop reads the buffer, dedupes by claim family (last observation
  wins), resolves supersede targets against the ACTIVE mindset, drops
  secret-poisoned captures individually, and publishes everything as
  ONE atomic thought via `micro_commit_batch` with the session's own
  author. On any refusal (e.g. someone's explicit staging in flight)
  the buffer survives and retries next turn — captures are never lost.
- session-start sweeps buffers older than 7 days (dead sessions).

## Acceptance criteria

- After PostToolUse the shared index is empty; after Stop it is still
  empty and the thought exists.
- Two interleaved sessions publish two separate thoughts, each with its
  own actor/author; neither absorbs the other's events.
- A dirty index defers publication without losing the buffer; the next
  Stop lands the captures.
- Existing capture semantics unchanged: supersede-on-new-value,
  identical-value dedupe, firehose mode, empty-turn no-op.
