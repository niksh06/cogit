# COG-063: project slug normalization + Aleph→aleph migration

Priority: P1 (field report: parallel sessions split one project into
`aleph` (230 beliefs) and `Aleph` (56 beliefs) — the qualifier is
identity-bearing, so the case variant was a silently separate family)

## Resolution (shipped with this ticket)

- `normalize_project_slug` in BOTH runtimes (identity parity): trim,
  lowercase, whitespace runs -> dashes. Applied at the single write
  choke point (`_write_fact_objects` / `write_fact_objects`) — every
  path (add-fact, micro-commit, batch, lifecycle, hook, MCP) inherits.
- Read filters normalize their input too (`_row_matches`/`row_matches`,
  health, analytics, hook COGIT_PROJECT env) — `--project ALEPH` finds
  `aleph`.
- Cookbook Rule 8 documents the slug contract; interop asserts the
  normalization agrees across runtimes.
- One-time live migration executed 2026-07-11: 56 `Aleph` beliefs
  re-homed to `aleph` in ONE atomic thought (originals removed with
  recorded reason `superseded`, ADR-0014), preserving actor /
  asserted_at / source / confidence / premises; method = `migration`.
  Post-check: Aleph=0, aleph=287; verify healthy.

## Caveat

Sessions whose MCP server predates this commit still write the raw
qualifier until restarted (same version-skew story as COG-066); the
migration is re-runnable if a stale writer strands more.

## Acceptance criteria

- Mixed-case/whitespace project input produces identical claim ids to
  the lowercase slug in both runtimes.
- Read filters are case-insensitive in facts/recap/dump/analytics/
  health/hook.
- Live journal holds zero `Aleph` beliefs; migrated rows keep original
  provenance and analytics classifies originals as superseded.
