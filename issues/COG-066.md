# COG-066: tolerant readers — additive must mean additive (ADR-0015)

Priority: P1 (field incident 2026-07-10: a session's long-running MCP
server, holding pre-ADR-0014 code, read a hook-written thought carrying
`removals` and died with `CorruptionError: unknown fields: ['removals']`
— reported by Sol/Fable during the aleph marathon)

## Root cause

Readers validated the FULL schema strictly on every `store.read`. Every
"additive" format extension (premises, removals) therefore broke every
deployed older reader the moment a newer writer touched the shared
journal. Additive was writer-only.

## Resolution (shipped with this ticket)

- `validate_object(mode="read")` / `validate_object_read`: unknown
  fields on KNOWN types are tolerated at read time in both runtimes;
  hash + canonical-bytes round-trip + all known-field checks still run;
  unknown object TYPES stay fatal. Writes remain fully strict.
- `verify` re-validates strictly and reports skewed objects as a
  WARNING (`unknown-fields`), so drift is visible without breakage.
- Invariant 5 refined; README design guarantees updated; ADR-0015
  records the git-headers precedent and the grandfathering caveat
  (readers older than this ADR still need a process restart — the only
  cure for already-deployed strict readers).

## Acceptance criteria

- A hash-valid object carrying an unknown field loads through
  `store.read` in both runtimes and preserves the field byte-for-byte.
- `store.write` still rejects unknown fields.
- An unknown object type is rejected even in read mode.
- Known-field violations (bad timestamp etc.) still fail on read.
- `verify` yields warning `unknown-fields`, zero errors, for a planted
  future object.
