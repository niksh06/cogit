# COG-068: `search` — cogit's git-grep over beliefs

Priority: P1 (field question: "как агенту искать по журналу, если он не
знает точный subject?" — filters were exact-match only; the gap between
exact ids and semantic RAG was empty)

## Resolution (shipped with this ticket)

`cogit search <pattern>` / MCP `search` in both runtimes:
case-insensitive substring over subjects, predicates, objects
(stringified), qualifier values and annotation bodies (detail notes are
searchable). Default corpus = the ref's ACTIVE beliefs; `--history`
widens to every assertion the ancestry ever held; every match carries
`active` and `matched_in`. `--project` scopes (case-normalized),
`--limit` caps detail with exact `total`/`truncated`. Output rows are
full fact rows — ids feed straight into `blame_fact`, premises walks
and family history.

Deliberate boundary: substring only — no regex (runtime-parity risk for
no proven need), no ranking, no embeddings (ADR-0002; the semantic
layer is COG-069/Engram).

## Acceptance criteria

- Match sets (ids, active flags, matched_in) identical across runtimes
  on one repository (interop step).
- Annotation bodies searchable; superseded beliefs findable only with
  `--history` and flagged inactive; empty pattern is a clean error.
