# COG-071: writer provenance on thoughts (ADR-0016)

Priority: P1 (owner decision 2026-07-12, pairs with COG-070)

## Problem

Looking at an old journal slice there is no way to tell which ERA of
cogit wrote it — pre-slug-normalization rows, thoughts without removal
provenance, etc. all look like anomalies instead of dated behavior. A
tool whose whole point is provenance of beliefs should carry provenance
of its own records. And with the version visible on artifacts, readers
can detect skew ("journal written by a newer cogit than me") instead of
failing obscurely.

## Resolution

- Optional `writer` field on **thought** objects: a single token
  `<impl>/<semver>`, e.g. `cogit-py/0.3.0`, `cogit-rs/0.3.0`; the
  format grammar reserves an optional `+<build>` suffix. Thoughts are
  the right (and only) carrier: they are the commit-analog and are not
  identity-deduplicated — claims/assertions MUST NOT carry it, or
  identical facts written by different versions would split into
  different IDs and break families/supersede.
- Written by every thought-creating path in both runtimes: staged
  `commit-thought`, merge thoughts, atomic micro-commits and
  `micro_commit_batch` (hence all lifecycle ops and `record`).
- Validation (write-strict, ADR-0015 read-tolerant as for any known
  field): if present — non-empty string, <= 64 chars, no whitespace or
  control characters, exactly one `/`.
- Golden vector #9: a thought carrying `writer` (frozen literal, does
  not track the live version). Earlier vectors unchanged.
- Exposure: `log --json` / `dump` log entries carry `writer` when the
  thought has it; `health()` reports `newest_writer` and warns
  (`version_skew`) when the journal holds thoughts written by a NEWER
  semver than the reader.

## Acceptance criteria

- A thought committed by Python carries `cogit-py/<v>`; read back by
  Rust byte-precisely (interop step), and vice versa.
- Malformed `writer` rejected on write in both runtimes; objects
  without `writer` (the entire existing history) stay valid.
- Golden vectors: all 9 reproduce byte-for-byte in both runtimes.
- `health()` on a journal containing a thought stamped with a higher
  semver than the reader emits a `version_skew` warning.

## Non-goals

- Stamping anchors/annotations (small overlays; a thought bounds every
  belief change already).
- Enforcing any ordering/monotonicity of writer versions in history —
  mixed-version journals are normal (parallel sessions, CLI vs MCP).
