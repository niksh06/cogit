# COG-072: viewer — project swimlanes + anchor chapters

Priority: P1 (owner pain, 2026-07-11: "мы видим одну ветку с кучей
якорей, но эти якори не подветки, и визуально сложно ориентироваться";
owner picked option B+A on 2026-07-12 from the four in
`~/Reports/projects/cogit/handoffs/cogit-tree-visualization-2026-07-11.md`)

## Problem

The git-graph renderer (COG-054) shows real branches — but a shared
journal is structurally LINEAR: one thread, 240+ thoughts, ~40 anchors
drawn as rings on a single lane. Anchors are tags, not branches, so
nothing reads as "the shape of the work". The user's mental model from
git — parallel lines of work — has no counterpart on screen, although
the DATA for it exists: every thought already carries the project
threads it touched (`node.projects`, COG-054).

## Resolution (client-side only; the protocol does not change)

**B. Project swimlanes** — a second graph layout, `projects`:
- every project is its own lane, ordered by activity, colored with the
  existing project color (chips and lanes agree);
- a thought's dot sits in the lane of its dominant (leftmost) project;
  thoughts with no project fall into a muted "journal" meta-lane;
- lanes are connected as continuous per-project threads (consecutive
  thoughts of the same project), and a multi-project thought spans its
  lanes with a horizontal connector plus satellite dots — visually the
  "олицетворение работ": parallel threads of parallel epics;
- this mode is a PROJECTION, not the DAG: parent edges are the
  `branches` mode's job, which stays exactly as it is one toggle away;
- view toggle in the Thoughts card: auto (>1 project → projects, else
  branches) / explicit; the choice is a URL param (`view=`), so a
  bookmarked view keeps its layout;
- lane legend above the graph: project chips in lane order with thought
  counts (click = existing project filter).

**A. Anchor chapters** — anchors partition the timeline:
- a segment = an anchored thought plus everything older, up to the next
  anchored thought; thoughts newer than the newest anchor form the
  "since ⚓ <newest>" head segment;
- segment headers are collapsible rows (chevron, anchor chips, thought
  count); default: head segment expanded, the rest collapsed — the
  journal opens as a table of contents of milestones;
- jumps into a collapsed segment (recap row, "Introduced by") expand it
  and scroll to the thought;
- journals without anchors render exactly as today (no headers).

Free side effect: collapsed segments shrink the DOM from N thought rows
to ~anchor-count rows — most of option D's windowing win without a new
view.

## Acceptance criteria

- Shared journal (multi-project) opens in swimlane mode with anchor
  chapters; branch-forked demo journal (README screenshot scenario)
  still opens in branch-lane mode unchanged.
- Toggling view or collapsing chapters never desyncs dots from rows
  (SVG rail and row list share one geometry).
- `?view=projects|branches` in the URL is honored and written back.
- Existing tests stay green; page carries the toggle; container
  rebuilt (viewer code is snapshotted into it).

## Non-goals

- Real branches in the working protocol (option C — separate decision).
- A textual `git log --graph` tab and true windowing (option D —
  later, if scale demands it).
- Any animation (sims-style rejected by owner 2026-07-09).
