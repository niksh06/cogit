# COG-073: agent-write ergonomics — close the model-vs-write-grain gap

Priority: P1 (owner goal 2026-07-12: "инфраструктура для агента, чтобы он
чувствовал себя как дома → максимальный КПД синергии человек+агент". Source:
first-hand dogfooding by an agent user (Fable) across a full engineering
session, 2026-07-12 — the agent used `add_fact` ~10× on live work and then
audited the read/nav surface.)

## The one number that frames everything

`lint(project=aleph)` on a real, actively-journaled project:
**412 findings / 305 warnings over 321 active facts.** Per rule:

| rule | count | what it means |
|---|---|---|
| R2-prose-object | 146 | `object` written as a sentence, not a value |
| R6-blob-qualifier | 107 | multi-field context stuffed into `qualifiers` |
| R3-subject-whitespace | 47 | subject is a human phrase, not a stable URI |
| R13-advisory-marker | 40 | lifecycle transition narrated in prose (not supersede/refute/retire) |
| R4-over/underconfident | 43 | confidence band vs kind mismatch |
| R10-generic-actor | 24 | non-specific actor |

This is **not one sloppy agent — it is the whole project, every agent.** When
~95% of real facts violate the claim-model, the conflict is not "agents are
lazy". It is a **structural gap between how an agent naturally writes in-flow
and cogit's ideal claim-model** — and the gap is paid **per write**, while the
correction (lint) arrives **after** the write.

## Why it happens (first-hand, in-flow)

An agent mid-task is a busy engineer, not a disciplined librarian. In the moment:

1. **`object` becomes prose (R2).** I narrate a finding as a sentence because
   that's how the thought forms. cogit wants terse value + nuance in `detail`.
   The tool fights the grain of how agents write.
2. **`qualifiers` become blobs (R6).** I dump status/verification/gates into
   qualifiers because it's one call and convenient. cogit wants separate claims.
3. **`subject` becomes a phrase (R3).** I write `"osv-distro full-scope land"`,
   not `cogit:osv-distro:land`. Slugging is friction I skip under load.
4. **Lint is post-hoc (timing).** I get told `R2-prose-object` in the `add_fact`
   RESPONSE — the debt is already committed. Retroactive cleanup in-flow won't
   happen; the agent has moved on.
5. **Lifecycle needs the old claim id (R13).** To `supersede_fact` I must first
   look up the prior assertion's id. Under load I skip it and just `add_fact` a
   new value → competing values (R11) + prose "changed to X" drift (the 40 R13).

Net: the beautiful analytics/query power (calibration, survival, families,
blame) rests on data the write-path makes hard to keep clean.

## What is already excellent (do NOT regress)

The git mental model is the killer feature and it transfers to an agent with a
**zero learning curve** — `add_fact`=commit, `dump`/`health`=re-anchor,
`supersede/refute/retire`=lifecycle, `branch/blame/log`. Keep all of it. Also
strong and unique: **`health`/`dump` one-call re-anchor** (perfect after a
compact), **atomic `commit=true`** (parallel agents, zero races — verified with
several agents on one journal today), **`analytics`** (an agent SEEING its own
calibration/survival is genuinely novel), **`search`+`blame_fact`+provenance**
(found my own fact instantly, full lineage). The bones are right. This issue is
about the write-path ergonomics only.

## Resolution — make the RIGHT thing the DEFAULT/CHEAP thing

Ordered by leverage. All must respect the no-LLM/deterministic ethos (ADR-0002):
in-server normalization is heuristic + suggestion, never a model.

1. **Lifecycle by (subject, predicate), not by id [kills R13/R11].**
   `supersede_fact(subject=…, predicate=…, object=<new>)` — cogit already knows
   the current active claim for that family; resolve the old id server-side.
   Removes the single biggest reason agents skip lifecycle → the 40 R13 + the
   R11 competing-values collapse. **Highest leverage, purely deterministic.**

2. **Write-time guardrail instead of post-hoc lint [fixes timing].** `add_fact`
   already computes the lint hints in the response. Escalate: when `object` is
   prose or a `qualifier` is a blob, return a **suggested normalized form**
   (`object`→terse + moved-to-`detail`; blob-qualifier→split) and either
   auto-apply with a flag, or make acceptance one cheap follow-up call. The
   agent corrects at write time, not never.

3. **Subject auto-slug [kills R3].** Whitespace subject → deterministically
   suggest/emit `project:area:entity` (lowercase, `:`-joined). Cheap, removes 47
   findings, and stabilizes ids for `facts`/`search`.

4. **"Journal as cheaply as a git commit message" [kills R2/R6 at the source].**
   The core UX target: the agent passes ONE natural line + optional hints;
   getting to a clean triple should feel like writing a commit message, not
   filling a structured form. Two ethos-safe paths:
   - **(a) client-side smart-journaling skill/prompt-guidance** (lives with the
     agent harness, not the server) that decomposes prose→subject/predicate/
     object/detail BEFORE calling `add_fact`. Keeps the server model-free.
   - **(b) heuristic server assist:** first clause→`object`, remainder→`detail`;
     detect predicate-like verbs. Crude but removes most R2 automatically.
   Recommend (a) as the real fix (a distributed "claim-modeling skill"), (b) as
   a cheap safety net.

5. **Ship the discipline as an agent-facing skill, not just docs.** `journal-
   discipline.md`/`claim-modeling.md` are excellent but are READ by humans. An
   agent needs the rules as an **inline, at-write affordance** (guardrail #2) + a
   loadable skill, not a doc it won't re-read mid-task.

## Root cause (one line)

cogit's model is correct and principled, but it asks the agent to be a
librarian at the exact moment it is an engineer; compliance cost is paid
per-write and corrected too late. Close the gap (lifecycle-by-family, write-time
guardrail, auto-slug, commit-message-cheap journaling) and cogit goes from
"principled but effortful" to "effortless AND principled" — the actual unlock
for agent adoption, and squarely the owner's goal #2 (agent feels at home →
human+agent КПД).

## Non-goals / guardrails

- **Do NOT weaken the model.** The triple + lifecycle + provenance discipline is
  what makes analytics/blame/families valuable. This is about making it CHEAP to
  comply, not optional.
- **Deterministic only in-server** (ADR-0002). Any prose→triple intelligence
  lives client-side (skill), or as transparent heuristics — never an LLM in the
  store.
- **Backward-compatible.** Existing facts + protocol unchanged; additions are
  new args/response fields + a client skill.

## Open questions

1. #1 (lifecycle-by-family): how to disambiguate when a (subject,predicate)
   family has >1 active claim (R11 already-dirty state) — refuse + point to lint?
2. #2 guardrail: auto-apply normalization vs suggest-and-confirm — which keeps
   agent trust without silent rewrites of its intent?
3. #4: is the client-side smart-journaling skill a cogit deliverable, or harness-
   side? (affects where the prose→triple logic is owned.)
4. Measure: re-run `lint` new-debt ratchet after shipping #1–#3 — target: new
   facts land <1 finding/fact vs today's ~1.3.
